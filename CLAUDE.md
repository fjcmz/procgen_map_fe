# CLAUDE.md

This file provides context and guidelines for AI assistants working on this codebase.

## Project Overview

Procedural fantasy map generator built with React, TypeScript, and Vite. Generates Voronoi-based terrain maps with biomes and rivers (always), partitions the terrain into geographic continents and regions (always), and optionally runs a full civilizational history (countries, wars, conquests, city placement, kingdom borders, roads) — all deterministic from a seed string.

**Key design principles**:
- The physical world (continents, regions, resources) is always built from terrain — it runs even without history.
- Cities and kingdoms are outputs of the history simulation, not independent pipeline steps. When history is disabled, the map shows terrain and geographic structure only.

## Commands

```bash
npm run dev       # Start Vite dev server (hot reload)
npm run build     # TypeScript type-check + Vite production build
npm run preview   # Serve the production build locally
npm run sweep     # Phase 4 seed sweep — runs full history across 5 fixed seeds in Node
                  # via tsx, writes scripts/results/<label>.json. Use `-- --label foo` to tag.
```

There is no test suite. Verify correctness by running `npm run build` (catches type errors) and visually inspecting the map in the browser. For history-simulation changes that might shift balance, run `npm run sweep -- --label <experiment>` and diff the resulting JSON against `scripts/results/baseline-a.json` to catch regressions. The sweep is deterministic — re-running with the same args produces byte-identical output modulo timestamps.

## Architecture

### Generation Pipeline

All heavy computation runs in `src/workers/mapgen.worker.ts` (a Web Worker). The pipeline is strictly sequential:

```
voronoi → elevation → oceanCurrents → moisture → temperature → biomes
  → fillDepressions → rivers (initial) → hydraulicErosion → fillDepressions
  → rivers (final) → temperature (refresh) → biomes (refresh)
  → buildPhysicalWorld
  └─ (if generateHistory=true) → HistoryGenerator → roads
                                    ├─ buildPhysicalWorld (World + Continents + Regions + Resources + Cities)
                                    ├─ TimelineGenerator (5000 years via YearGenerator)
                                    │    └─ 12 Phase 5 generators per year:
                                    │       Foundation → Contact → Country → Illustrate → Religion
                                    │       → Trade → Wonder → Cataclysm → War → Tech → Conquer → Empire
                                    └─ serialize → HistoryData (ownership snapshots + events)
```

The terrain steps (voronoi through rivers + hydraulic erosion) always run. Before each river trace a **Priority-Flood + ε depression-fill pass** (`fillDepressions`) builds a parallel `drainageElevation: Float32Array` so every land cell has a strictly-lower neighbour toward the ocean — this guarantees rivers never dead-end mid-land. The pass also materializes any closed basin of size ≤ `profile.lakeMaxSize` (default 60 cells) as a LAKE biome: affected cells flip to `isWater = true` / `isLake = true` / `biome = 'LAKE'`. Larger basins stay as land and rely on the virtual `drainageElevation` surface to route rivers through. After rivers are initially traced, hydraulic erosion carves valleys using a stream power model (erosion proportional to flow^0.5 × slope), with sediment deposition creating floodplains and valley widening for hillshading visibility; erosion deliberately uses raw `cell.elevation` (its deposition step breaks monotonicity, so a cached `drainageElevation` would go stale). `fillDepressions` runs a second time on the eroded terrain before the final river trace. Rivers are then re-traced on the carved terrain for precise valley-following paths. Temperature and biomes are refreshed after erosion since they depend on elevation (lapse rate, elevation bands); the `isLake` short-circuit in `assignBiomes` protects LAKE cells from being overwritten by the refresh, but it runs *after* the polar ICE block so cold inland lakes can still freeze. Moisture and ocean currents are NOT re-run (they depend on coarse geography, not fine elevation detail). Elevation uses tectonic plate simulation with controlled continental/oceanic split (3–5 large continental plates clustered via nearest-neighbor seeding, 8–12 oceanic plates spread via farthest-point sampling; size-biased round-robin BFS gives continental plates ~3× more cells; continental seam elevation boost merges adjacent continental plates; convergent/divergent boundary effects, polar ice caps with smoothstep blending for gradual transitions, thermal erosion). Ocean currents run after elevation: BFS flood-fill identifies connected ocean basins, then an analytical gyre model computes per-cell SST anomalies (warm poleward currents on western basin margins, cold equatorward currents on eastern margins, scaled by latitude envelope). Moisture applies three layers: base FBM + smooth Hadley cell latitude curve (damped cosine modeling three atmospheric cells per hemisphere — wet equator, dry subtropics, moderate midlatitudes, dry poles) + coastal boost (modulated by ocean currents — cold currents suppress evaporation for drier eastern-margin coasts), then continentality gradient (BFS distance-from-ocean decay for dry interiors), then rain shadow (upwind mountain barrier march using prevailing wind simulation). Temperature is computed from latitude + continentality modifier (continental interiors more extreme, maritime cells milder) + windward ocean proximity (west-coast effect via upwind march through neighbor graph) + ocean current influence (SST anomaly of upwind ocean cells propagated to nearby land) + elevation lapse rate + noise perturbation; water cells incorporate SST anomaly directly; it drives the polar biome thresholds and nudges the Whittaker moisture lookup at margins. `buildPhysicalWorld` also always runs — it annotates cells with `regionId`, and produces `RegionData[]`/`ContinentData[]` in `MapData` regardless of the history flag. The history simulation is opt-in via `GenerateRequest.generateHistory`. When enabled, the **HistoryGenerator** (Phase 6) orchestrates the full pipeline: it calls `buildPhysicalWorld` to create the physical world hierarchy, then `TimelineGenerator` to run 5000 years of simulation, then serializes the result into the flat `HistoryData` format for the UI. If history is disabled, the pipeline ends after `buildPhysicalWorld` — no kingdom simulation, roads, or timeline data is generated.

Each step is a pure function in `src/lib/` that takes cells and returns updated cells or derived data.

### City Map V2 (Voronoi-polygon foundation)

The Details tab has two city-map buttons: **Map** (V1, tile-based) and **MapV2** (V2, Voronoi-polygon-based). They are intentionally coexisting during a phased migration tracked in `specs/City_style_phases.md`.

V2 replaces the V1 tile grid with a Voronoi polygon graph sized by city tier: small 150, medium 250, large 350, metropolis 500, megalopolis 1000. The polygon graph (`CityPolygon[]` in `CityMapDataV2.polygons`) is the single data contract PR 2-5 build on top of — walls are traced along polygon edges (PR 2); rivers, roads, and streets route over the polygon edge graph (PR 3); blocks are flood-fills of the polygon adjacency graph (PR 4); buildings are packed per polygon (PR 5). Each polygon carries `vertices` (unclosed ring), `neighbors` (Delaunay adjacency), `isEdge` (touches canvas bbox — used by PR 2 wall footprint), and `area` (shoelace, used by PR 4 landmark placement and PR 5 label sizing).

**Current PR 5 status (buildings + outside-walls sprawl slices landed; all of PR 4 already landed)**: polygon graph + flat-paper base + faint polygon-edge cadastral grid + city-name label (PR 1); walled-blob rendering with 0–4 cardinal gates per city (PR 2, `cityMapWalls.ts` + renderer Layer 11); river + roads + streets + bridges (PR 3, `cityMapRiver.ts` + `cityMapNetwork.ts` + shared `cityMapEdgeGraph.ts`, renderer Layers 5–8); AND the open-spaces slice of PR 4 (`cityMapOpenSpaces.ts` + renderer Layer 9); AND the blocks slice of PR 4 (`cityMapBlocks.ts`, data-only — district labels that consume block geometry are PR 5); AND the landmarks slice of PR 4 (`cityMapLandmarks.ts` + renderer Layer 12); AND the buildings slice of PR 5 (`cityMapBuildings.ts` + renderer Layer 10); AND the outside-walls sprawl slice of PR 5 (`cityMapSprawl.ts` + renderer Layer 4). `wallPath`, `gates`, `river`, `bridges`, `roads`, `streets`, `openSpaces`, `blocks`, `landmarks`, `buildings`, and `sprawlBuildings` are now populated. The river traces polygon edges between two non-adjacent canvas sides (boundary vertices come from `isEdge` polygon rings, not raw canvas-bbox vertices); large+ cities attempt one bifurcation per river (35% chance per attempt, up to 6 attempts, middle-third stretch of the main path, interior vertices blocked so the alternate route is guaranteed disjoint, locality-bounded to ~2.5 × avg-edge-length from the stretch); island polygons are detected by a polygon-graph flood seeded from every `isEdge` polygon with river edges treated as walls (catches multi-polygon islands that a ring-of-edges check would miss). Roads A* from each gate's nearest polygon vertex to the polygon vertex nearest canvas center, with a normalized-dot turn penalty, moderate river-crossing cost, and small per-step noise. Streets space-fill the polygon graph until every interior polygon is "served" (has ≥1 ring edge in the covered set) or has a served Delaunay neighbor — translating the spec's "within ≤1 tile" to polygon adjacency. Bridges are the canonical-edge-key intersection of road edges and river edges (fully deterministic). Open spaces are polygon-keyed (`{ kind, polygonIds }[]`): one deterministic **civic square** on the polygon nearest canvas center; `MARKET_COUNT[size]` **markets** (`small:2, medium:4, large:5, metropolis:8, megalopolis:16`) anchored to the polygons nearest each gate midpoint, with remaining slots filled by Lloyd-style `farthestPolygonBySite`; `PARK_COUNT[size]` **parks** (`small:1, medium:1, large:2, metropolis:2, megalopolis:3`) grown by `polygon.neighbors` BFS from seed polygons that avoid Delaunay neighbors of already-used polygons (cluster cap `PARK_MAX_POLYGONS[size]`, 1–3 polygons). Eligibility excludes `polygon.isEdge` and any polygon whose vertex ring shares a canonical edge key with the wall / river / road paths; streets are not excluded so plazas can front them. Blocks are a partition over every `polygon.id`: a polygon-graph flood-fill bounded by wall / river / road / **street** canonical edge keys (streets ARE barriers, unlike open-space eligibility) PLUS an implicit footprint-boundary barrier driven by `wall.interiorPolygonIds` membership (so interior and exterior polygons never end up in the same block even when the city has no walls); each connected component becomes one `CityBlockV2` with a role (civic/market from openSpaces polygons, harbor from `env.waterSide` proximity, slum/agricultural when the block's polygons sit outside the footprint, else residential) and a medieval `PREFIX+SUFFIX` name ported from V1's flavor word lists. Landmarks are single-polygon anchors sourced from civic/market block roles: capitals get castle and/or palace on the civic polygon(s) nearest canvas center (small/medium → one via RNG coin-flip, large+ → both), one temple per `env.religionCount` picked randomly from the broad civic+market pool, one monument per `env.wonderCount` picked from a hybrid pool that favors plaza polygons via a V1-parity duplicate-concat weighting. Three landmark RNG sub-streams (`_landmarks_capitals` / `_landmarks_temples` / `_landmarks_monuments`) fan off the shared `${seed}_city_${cityName}_` prefix. Buildings are polygon-interior rejection-sampled rects (4–12 per non-reserved interior polygon, role-driven size bands, 1 px mortar, mix of solid / hollow `#2a241c` ink) keyed on `CityBuildingV2.polygonId`; the `_buildings` RNG sub-stream fans off the same shared prefix. Sprawl is the mirror-image slice: the same rejection-sampling recipe applied to `slum`/`agricultural` blocks (0–4 rects per polygon, smaller size bands for huts/farmhouses, `SPRAWL_TIER_SCALE` multiplier so megalopolis fringes read busier than small-city fringes, same `#2a241c` ink but `SPRAWL_STROKE_WIDTH = 0.6` for an airier read), serialized as a separate `sprawlBuildings: CityBuildingV2[]` field so the renderer can draw it on Layer 4 distinct from Layer 10 interior buildings. The `_sprawl` RNG sub-stream is independent from `_buildings` so each slice's seed stability is preserved when future PR 5 slices (`_docks`, `_labels`) land. The remaining `CityMapDataV2` field (`districtLabels`) still stays empty with a `// TODO PR 5 (remainder):` marker. The rest of PR 5 — **dock** hatching along `env.waterSide` and rotated **district labels** ("BLUEGATE", "GLASS DOCKS", …) — lands in follow-up slices; only after all of PR 5 is in do we retire V1 (`cityMapGenerator.ts`, `cityMapRenderer.ts`, `CityMapPopup.tsx`, the "Map" button) and unify the popups.

**Invariants**:
- **Do NOT reintroduce tiles under `src/lib/citymap/`** — V2 is polygon-based by design; the pivot away from tiles is called out in every V2 file header so PR 2-5 authors see it immediately. The spec's V1 tile language ("tile count", "1–2 tile offset", "within ≤1 tile") is translated to polygon/edge language inside each generator; keep spec quotes inside file headers, not inside code.
- **No `Math.random` anywhere under `src/lib/citymap/`** — always route through `seededPRNG` from `terrain/noise.ts`. The V2 generator uses a dedicated `_voronoi` suffix on the polygon-graph stream so PR 2-5 RNG streams stay decoupled and seed-stable across PR landings. PR 2 added `_walls` (`createNoiseSamplers(${seed}_city_${cityName}_walls)` fed into `fbm` for the wall-footprint perturbation); PR 3 added `_river` (river meander + side/endpoint choice), `_roads` (road per-step noise), and `_streets` (polygon pick + vertex pick). PR 4 (open-spaces slice) added `_openspaces_markets` (Lloyd-style spread fallback + tie-breaking) and `_openspaces_parks` (seed pick + cluster size); the renderer also fans off `_openspaces_render_markets` and `_openspaces_render_parks` for stall / tree scatter so re-rendering without re-generating doesn't perturb generation output. PR 4 (blocks slice) added `_blocks_names` — the only RNG use in the blocks pipeline, since the polygon-graph flood and role classification are fully deterministic from geometry. PR 4 (landmarks slice) added `_landmarks_capitals` / `_landmarks_temples` / `_landmarks_monuments` so future landmark kinds can be inserted without shifting existing seeds. PR 5 (buildings slice) added `_buildings` — single stream keyed on `${seed}_city_${cityName}_buildings`; iteration order inside the generator is fixed (block order → polygon.id order → slot-by-slot rect rolls) so future PR 5 streams (sprawl / docks / labels) cannot perturb the building layout regardless of insertion order. PR 5 (sprawl slice) added `_sprawl` — independent of `_buildings` with its own fixed iteration order (block order → polygon.id order → slot-by-slot), so the two polygon-interior packers are seed-stable both in isolation and when run together. Bridge detection and the civic-square pick are fully deterministic and use no RNG. All PR 3+ streams fan off the shared `${seed}_city_${cityName}_` prefix.
- **Polygon-edge graph helpers are centralized** — `roundV`, `vertexKey`, `canonicalEdgeKey`, `EdgeRecord`, `buildEdgeOwnership`, `buildPolygonEdgeGraph`, `aStarEdgeGraph`, `keyPathToPoints`, and `nearestVertexKey` all live in `src/lib/citymap/cityMapEdgeGraph.ts`. `cityMapWalls.ts`, `cityMapRiver.ts`, and `cityMapNetwork.ts` import from there; PR 4+ feature files must do the same rather than re-declaring float-tolerance-sensitive helpers in their own modules (inconsistent `VERTEX_PRECISION` values would silently break shared-edge collapse and index the same edge under two different keys).
- `generateCityMap` / `renderCityMap` (V1) and `generateCityMapV2` / `renderCityMapV2` (V2) signatures are frozen — only the V2 data shape evolves.
- V2 Lloyd relaxation runs 2 rounds (mirrors `terrain/voronoi.ts`); 3+ rounds over-regularize the graph and start to look grid-like, defeating the organic look.
- `buildCityPolygonGraph` strips the `last===first` closing vertex returned by D3's `cellPolygon` so `polygon.vertices` is always unclosed; PR 2-5 assume unclosed rings.
- **River + roads + streets share one polygon-edge graph** — `cityMapGeneratorV2.ts` builds `buildPolygonEdgeGraph(polygons)` once after walls, then hands the same `PolygonEdgeGraph` to `generateRiver` and `generateNetwork`. Don't rebuild it per feature; precomputed `edgeLen` and `avgEdgeLen` are how the A* cost functions stay dimensionally consistent (e.g. `rng() * 0.4 * avgEdgeLen` for river meander, `ROAD_TURN_PENALTY_COEFF * avgEdgeLen` for road turn penalty).
- **River bifurcation must produce a disjoint detour** — the A* alternate path blocks the stretch's interior vertex keys via `aStarEdgeGraph`'s `blockedKeys` option; do NOT try to just remove the stretch edges and re-run A*, because A* could still snake back through the same interior vertices via other edges and produce a 1-vertex spike instead of an island.
- **Island detection must use polygon flood, not ring-check** — `floodDetectIslands` in `cityMapRiver.ts` floods `polygon.neighbors` from every `isEdge` polygon with river edges treated as walls; polygons not reached are islands. A ring-of-river-edges check only finds single-polygon islands and misses multi-polygon islands the bifurcation naturally creates.
- **Street "served" semantics** — a polygon is served iff it has ≥1 ring edge in `coveredEdges = wallEdges ∪ roadEdges ∪ riverEdges ∪ streetEdges`. A polygon is an eligible street-frontier iff it's not served AND no Delaunay neighbor is served (the polygon-graph analog of V1's "within ≤1 tile"). `isEdge` polygons are never frontier candidates — they belong to PR 5's outside-walls sprawl.
- **Open-space eligibility excludes infrastructure-touching polygons, NOT street-touching polygons** (PR 4 slice) — `cityMapOpenSpaces.ts` builds a single `blockedEdgeKeys: Set<string>` covering every wall / river / road segment via `canonicalEdgeKey`, then filters out any polygon whose vertex ring shares one of those keys (plus all `polygon.isEdge` polygons, which belong to PR 5 sprawl). Streets are deliberately omitted from the blocked set — plazas, markets, and small parks should front streets to read as "lived-in" spaces, not isolated cul-de-sacs. If you add new infrastructure that should reject open-space placement (e.g. PR 5 docks), add its edges to that same `blockedEdgeKeys` set rather than walking polygon-by-polygon. Iteration order inside the generator is fixed (civic → markets → parks) so the `used` set grows in a predictable sequence; flipping that order would shift which polygons end up in which kind on every seed.
- **Block barriers INCLUDE streets — semantic inversion vs. open-space eligibility** (PR 4 slice) — `cityMapBlocks.ts` builds its own `barrierEdgeKeys: Set<string>` covering every wall / river / road / **street** segment. Streets are the primary boundary between urban districts (blocks are the "bounded-by-streets" flood-fill the spec describes at line 63), so they must be in the barrier set even though openSpaces deliberately omits them. Future feature files under `src/lib/citymap/` must NOT copy `cityMapOpenSpaces.ts`'s `blockedEdgeKeys` verbatim when they mean "block barrier"; the two sets have different semantics and are intentionally out of sync. Block membership is a strict partition: every polygon in `polygons` appears in exactly one `block.polygonIds`. The interior/exterior split is driven by `wall.interiorPolygonIds` (the organic city footprint allocated by `cityMapShape.ts`), NOT by `polygon.isEdge` — blocks whose polygons sit outside the footprint become `slum`/`agricultural` depending on cluster size vs. `SLUM_SIZE_THRESHOLD[env.size]`, regardless of whether the city is walled. This decoupling is the mechanism that lets small/medium unwalled cities still produce civic/market/harbor/residential blocks inside the footprint (previously they rendered empty because the flood merged interior polygons with canvas-edge polygons and every block got tagged exterior). Role assignment precedence is `slum/agricultural > civic > market > harbor > residential` — exterior short-circuits first so an open-space polygon outside the footprint (rare but possible when openSpaces runs with a degenerate footprint) never taints an agricultural cluster with a civic tag. The `_blocks_names` RNG sub-stream is the only RNG fan-out; the flood + role classification are deterministic from geometry, so seed-stability is preserved end-to-end.
- **Landmarks anchor to one polygon, never two** (PR 4 slice) — `cityMapLandmarks.ts` runs three ordered passes (capitals → temples → monuments) that share a single `used: Set<number>` for de-duplication. Once a polygon is used by any pass, no later pass can place on it. Candidate pools are sourced from block ROLES (`civic` / `market`), NOT from raw polygon geometry — the block layer is the canonical "where can landmarks go?" authority. When extending (e.g. adding guildhalls or shrines), reuse the block-role filter and extend the three-pass structure; do NOT reach directly into `polygons` or `openSpaces` without going through the block pool. Pass order is fixed (capitals first so the most-central civic polygon is reserved for castle/palace, then temples from the broad pool, then monuments from the hybrid plaza-weighted pool) — flipping the order would shift which polygon each landmark lands on for every seed. The monument hybrid pool intentionally duplicates plaza polygons across the concat (`[...squarePool, ...civicAndMarket]`) to give them ~2× random-pick weight; treat that concat as part of the seed-stable contract and do not dedupe it.
- **V2 landmark glyphs are sized from `√polygon.area`, not `tileSize`** (PR 4 slice) — `cityMapRendererV2.ts::drawLandmarks` derives glyph bounding-box size as `clamp(20, 36, √polygon.area × 0.7)` and centers on `polygon.site`. V2 has NO tile concept, so V1's `tileSize` is gone entirely. The clamp window keeps small-tier cities (150 polygons ⇒ larger average area) and megalopolis-tier cities (1000 polygons ⇒ smaller average area) both producing legible silhouettes. If a future change wants variable-size wonders (spec line 13: "higher tier wonders appear bigger"), thread a `tier?` field into `CityLandmarkV2` at the generator layer rather than re-reading `env.wonderCount` from the renderer — the renderer is intentionally RNG-free and data-driven. Render order: Layer 12 landmarks are drawn AFTER Layer 11 walls so capital silhouettes sit on top of wall ink.
- **Buildings are polygon-interior rejection-sampled, NOT edge-graph traversed** (PR 5 slice) — `cityMapBuildings.ts::generateBuildings` rolls candidate rects inside each eligible polygon's bounding box and accepts only those whose four corners (a) all pass `pointInPolygon(corner, polygon.vertices)` AND (b) all sit ≥ `INSET_PX = 2` px from every polygon edge. Voronoi cells are convex, so all-four-corners-inside ⇒ whole rect inside; the distance-to-edge check replaces an explicit polygon-inset ring and keeps buildings from kissing streets / roads / walls (which all live on polygon edges). A secondary AABB overlap test with `MORTAR_PX = 1` padding enforces the spec's 1 px mortar gap. Because this is interior-packing geometry, `cityMapBuildings.ts` INTENTIONALLY does NOT import `cityMapEdgeGraph.ts` — there's no canonical-edge-key matching, no A*, no edge ownership. When extending (e.g. adding chimneys, courtyards, variable-pitch roofs), keep the polygon-interior-only discipline; reach for the edge graph only if the new feature needs to align to polygon edges (like future sprawl might, to lean buildings against walls). Render: Layer 10 is drawn between bridges (Layer 8) and walls (Layer 11) so buildings cover the cream base but walls / landmarks sit visibly on top of any rect touching them; hollow strokes are inset by half the `lineWidth` so the 1 px mortar gap reads crisply. No render-time RNG — every dimension is baked into `CityBuildingV2`.
- **Buildings skip slum / agricultural blocks by design** (PR 5 slice) — `PACKING_ROLES` in `cityMapBuildings.ts` is `{'civic', 'market', 'harbor', 'residential'}`. `slum` and `agricultural` blocks live on `polygon.isEdge`-containing clusters and belong to the outside-walls sprawl slice of PR 5 (sparse scatter, different style, different RNG stream, different generator). The `reserved` set passed to `generateBuildings` is also precisely `openSpaces ∪ landmarks` — nothing else — so adding future reserved kinds (e.g. cemeteries, guildhall yards) must be routed through either `openSpaces` or `landmarks` rather than a third set, otherwise the building-eligibility check silently drifts out of sync with the visual layer semantics.
- **Sprawl inhabits exactly what buildings skip** (PR 5 sprawl slice) — `cityMapSprawl.ts::generateSprawl` targets `SPRAWL_ROLES = {'slum', 'agricultural'}`, the complement of `cityMapBuildings.ts::PACKING_ROLES`. The two generators share the same `reservedPolygonIds = openSpaces ∪ landmarks` composition so plazas, parks, and landmark glyphs are never overwritten regardless of whether they happen to land on interior or fringe polygons. Output is a separate `sprawlBuildings: CityBuildingV2[]` field (NOT a mutation of `buildings`) and rendered on **Layer 4** via a dedicated `drawSprawl` helper — do NOT fold sprawl rects into `buildings` to save a field: the separate layer lets future tuning change the sprawl ink / stroke / scatter style independently. Sprawl iterates the WHOLE block (every polygon in every `slum` / `agricultural` block), not a per-polygon `isEdge` filter — because `cityMapBlocks.ts::isExteriorBlock` now classifies by `wall.interiorPolygonIds` membership (a block is exterior iff its polygons sit outside the footprint), exterior clusters include any non-footprint polygons caught between the footprint boundary and the canvas edge and we sprawl over all of them so the fringe reads as one continuous cluster rather than a ring of dots on the canvas border; the block role is the authoritative "outside-walls" tag. Dedicated RNG sub-stream `${seed}_city_${cityName}_sprawl`, independent of `_buildings`. Tier scaling via `SPRAWL_TIER_SCALE` (0.5–1.5) addresses spec line 23 ("the bigger the city the more such sparse buildings"); the per-polygon clamp is `[0, 4]` so even megalopolis fringes stay visibly sparser than interior packing's `[4, 12]`. Polygon-interior geometry helpers (`pointInPolygon`, `distanceToPolygonEdge`, `pointSegmentDistance`, `rectsOverlapWithMortar`) are DUPLICATED from `cityMapBuildings.ts:319-385` — each V2 slice keeps its own geometry helpers (same convention as `cityMapLandmarks.ts` / `cityMapOpenSpaces.ts`); if you later need to factor them out, do the buildings and sprawl files together so the inset / mortar math stays byte-identical between the two polygon-interior packers. `cityMapSprawl.ts` INTENTIONALLY does NOT import `cityMapEdgeGraph.ts` — sprawl is polygon-interior rejection sampling, not edge-graph traversal, same discipline as buildings. Render order: Layer 4 is drawn BEFORE the river / streets / roads / walls passes so any future gate-exiting road (spec line 24: "roads going from the gates to the outer limits of the map") will visibly sit on top of sprawl it crosses.
- `npm run sweep` is unaffected by citymap changes (city-map generation is render-only, never reached by the sweep harness). Any non-zero sweep diff after a citymap-only change means an accidental simulation-layer edit.

### Data Model

The central type is `Cell` (defined in `types.ts`). Every terrain step annotates cells with new fields:

- `elevation`, `moisture` → set by `terrain/elevation.ts` / `terrain/moisture.ts`
- `temperature` → set by `terrain/temperature.ts` (0 = coldest polar extreme, 1 = hottest equatorial extreme; accounts for latitude, continentality, windward ocean proximity, and elevation lapse rate)
- `biome` → set by `terrain/biomes.ts`
- `river`, `flow` → set by `terrain/rivers.ts`
- `isLake` → set by `terrain/depressionFill.ts`; `true` for cells materialized as inland lakes from small closed basins (also flips `isWater = true` and forces `biome = 'LAKE'`). `assignBiomes` short-circuits on this flag *after* the polar ICE check, so cold lakes can still freeze into ICE
- `regionId` → set by `history/history.ts` (`buildPhysicalWorld`), always present after generation
- `kingdom` → set by `history/history.ts` (year-0 BFS, updated by renderer at selected year)

`Season` (0–3): render-time-only type controlling seasonal ice/snow/tundra boundary shifts and permafrost overlay. 0 = Spring (baseline, no change), 1 = Summer (ice retreats), 2 = Autumn (slight expansion), 3 = Winter (maximum expansion). Not stored on cells — applied at draw time via `getSeasonalBiome()` and `getPermafrostAlpha()`.

`PoliticalMode` (`'countries' | 'empires'`): render-time sub-toggle active only when `mapView === 'political'`. `'countries'` fills each country's cells with a unique diagonal stripe pattern (two deterministic colors per country). `'empires'` fills empire-member cells with a plaid/tartan pattern (two deterministic colors per empire); independent countries retain their diagonal stripes. Colors are generated via golden-ratio hue spacing from entity indices. Managed by `politicalMode` state in `App.tsx`, passed through `MapCanvas` → `render()`.

`MapData` (returned from worker) carries:
- `cells` — always present; each cell has `regionId?` after generation
- `regions?`, `continents?` — always present (built even without history); serializable `RegionData[]`/`ContinentData[]` for rendering geographic structure
- `cities?`, `roads?` — only present when history was generated
- `history?` — `HistoryData` with `countries`, `years[]`, decade `snapshots`
- `historyStats?` — `HistoryStats` aggregate (Phase 3): peak population, totals per event type, plus `totalTechs` and `peakTechLevelByField` for tech introspection. **Phase 4** adds `totalTrades`, `totalConquests`, `totalCataclysmDeaths`, `techEventsPerCenturyByField` (tech events bucketed by 100-year window per field), `peakCountryTechLevelByField`, and `medianCountryTechLevelByField` — used primarily by the `scripts/sweep-history.ts` balance harness but available on every run at zero extra cost. **Spec stretch §1** adds `totalTechLosses` and `totalTechLossesAbsorbed` for cataclysm-driven knowledge regression tracking. Forwarded straight from `historyGenerator.generate()`; do not recompute on the main thread.

`HistoryData` structure:
- `countries: Country[]` — each country has id, name, capitalCellIndex, isAlive
- `years: HistoryYear[]` — per-year events (15 types: Foundation, Contact, Country, Illustrate, Wonder, Religion, Trade, Cataclysm, War, Tech, Conquest, Empire, plus legacy Merge/Collapse/Expansion) + sparse `ownershipDeltas`
- `snapshots` — full `Int16Array` of cell→countryId at every 20th year (for fast scrubbing)
- `tradeSnapshots` — `Record<number, TradeRouteEntry[]>` snapshotted every 20 years; each `TradeRouteEntry` has `cell1`, `cell2`, and an optional `path: number[]` (A*-pathfound cell-index sequence that hugs coastlines for maritime routes; absent only as fallback)
- `empireSnapshots` (overlay Phase 4) — `Record<number, EmpireSnapshotEntry[]>` snapshotted every 20 years + final year, aligned with `snapshots` cadence so both lookups use the same `Math.floor(year / 20) * 20` formula. Each entry carries `empireId`, display `name` (generated from 54 diverse title templates like "Kingdom of {name}", "The {name} Dominion", etc., using the founder country's generated name; cached per empire so the name persists across snapshot years), `founderCountryIndex`, and sorted `memberCountryIndices` — all projected through `countryMap.idToIndex` so the UI can render a stable Empire → Country → City tree without re-walking events. The payload is additive: `HistoryStats` is untouched, so `npm run sweep` output is byte-identical
- `techTimeline?` (spec stretch §5) — `{ byField: Record<TechField, Uint8Array> }` per-field running-max tech level indexed by year offset (0..numYears-1). Computed inline with the `peakTechLevelByField` walk in `HistoryGenerator.generate()` — NOT a second pass over `timeline.years`. Each array has length `yearsToSerialize` (matches `numYears`), and writes inside the main loop are guarded with `if (yi < yearsToSerialize)` so truncated runs don't silently drop tail data. After the write loop a cheap forward-fill pass propagates each year's max forward through quiet years so the polylines stay flat instead of dropping to zero. Monotonic by design: `TECH_LOSS` events do NOT dip the curves (the chart shows "highest peak ever attained", not "current global level"). Payload is ~9 × numYears bytes (45 KB at default 5000-year runs) — safe across `postMessage` as `Uint8Array` is structured-clone safe, same family as `snapshots: Record<number, Int16Array>`

`HistoryEvent` Phase 3 enrichments (all optional):
- `discovererType` — illustrate type for TECH events (`'science' | 'military' | 'philosophy' | 'industry' | 'religion' | 'art'`)
- `countryName` — resolved country display name for TECH and CONQUEST events
- `acquiredTechs` — `{field, level, displayName?}[]` delta the conqueror gained, populated on CONQUEST events only when non-empty. Spec stretch §3 adds the optional `displayName` flavor string to each entry
- TECH event descriptions are now formatted by `buildTechDescription()` in `HistoryGenerator.ts` as `"{Country} discovers {displayName} ({field} L{level}) (by a {illustrateType} in {city})"` with graceful fallbacks for stateless / displayName-less techs (spec stretch §3)
- CONQUEST descriptions append `(+N techs)` when the delta is non-empty
- `lostTechs` / `absorbedTechs` (spec stretch §1) — `TECH_LOSS`-only structured payloads carrying the per-country tech delta from a knowledge-destroying cataclysm. `lostTechs` is `{field, newLevel}[]` (post-decrement; `newLevel === 0` means the field was removed); `absorbedTechs` is `{field, level}[]` for rolls that `government >= 2` silently absorbed. One `TECH_LOSS` event is emitted per affected country per cataclysm
- `displayName` (spec stretch §3) — TECH-only flavor name resolved at serialization time via `nameForLevel(field, level)` from `timeline/techNames.ts`; appears on the TECH event and on each entry of CONQUEST `acquiredTechs`. Purely cosmetic, never affects simulation state
- `propagationReason` (spec stretch §4) — RELIGION-only optional enum `'art' | 'government' | 'both'` (`'none'` is omitted entirely, not stored). Computed at serialization time in `HistoryGenerator.ts` from the religion's `originCountry` current tech state via `getCountryTechLevel('art')` and `getCountryTechLevel('government')`. When present, the event description gets a trailing `(spread boosted by art/government/art and government)` suffix via the same enrich-description pattern used by `techDiffusion`. Read-side only: does not affect simulation state and is safe to toggle serialization logic without a sweep rerun

### Physical World Model

`buildPhysicalWorld(cells, width, rng)` in `history/history.ts` always runs before the optional history simulation. It uses the Phase 3 generator classes internally:

1. **Continents**: BFS flood-fill finds connected land cells; groups ≥ 10 cells form a `Continent` (via `continentGenerator`)
2. **Regions**: each continent is subdivided into ~30-cell clusters via multi-source BFS seeding; each gets a `RegionBiome` derived from its dominant Voronoi biome (via `regionGenerator`); geographic adjacency is wired with `regionGenerator.assignNeighbours`; `regionGenerator.updatePotentialNeighbours` computes BFS-layered `potentialNeighbours` (distance graph) for all regions after all continents are built
3. **Resources**: 1–10 `Resource` entities per region, weighted-random type (17 types: strategic/agricultural/luxury) via `resourceGenerator`
4. **Cities**: 1–5 `CityEntity` objects per region, placed on highest-scoring terrain cells via climate-aware `scoreCellForCity`, via `cityGenerator` (which also inserts into `world.mapCities`). City names are procedurally generated via `generateCityName()` from `nameGenerator.ts` using a syllable combinator (onset+vowel+coda, 2–3 syllables, with optional fantasy suffixes like `-vale`, `-holm`, `-gate`); a `Set<string>` ensures no duplicate names within a generation. Scoring boosts river cells (tiered by flow: >4/15/40), river mouths (coast+river), natural harbors (coastal cells with ≥4 land neighbors), and penalizes extreme biomes (tundra -5, desert -4, bare/scorched -3, temperate desert/marsh -2) with mitigation from rivers (-2) and coast (-1) so harsh-biome cities still appear near water features

The generator singletons (`worldGenerator`, `continentGenerator`, `regionGenerator`, `resourceGenerator`, `cityGenerator`) encapsulate object creation and map-insertion. The visitor singletons (`cityVisitor`, `regionVisitor`) provide iteration and predicate-based selection over the world's runtime index maps — used by Phase 4+ timeline simulation.

The `World`/`Continent`/`Region`/`CityEntity`/`Resource` class instances live **only inside the worker** — they use `Map`/`Set` which are not structured-clone safe and cannot cross the `postMessage` boundary. The worker serializes them into plain `RegionData[]` and `ContinentData[]` arrays for `MapData`.

`CityEntity` (in `physical/CityEntity.ts`) is the rich simulation entity tracking full lifecycle state. It is distinct from the lightweight render-type `City` in `types.ts`, which is used by the renderer for icon/label drawing.

### Timeline Model

`Timeline` and `Year` (in `history/timeline/`) form the temporal simulation layer (Phase 4) that runs on top of the physical world. The `TimelineGenerator` creates a `Timeline` anchored at a random start year ([-3000, -1001]) and generates 5000 `Year` records via `YearGenerator`.

Each `Year` holds 12 typed event collections (foundations, contacts, countries, illustrates, wonders, religions, trades, cataclysms, wars, techs, conquers, empires) populated by the Phase 5 generators during year generation.

`YearGenerator.generate()` performs 9 preprocessing steps per year, then calls all 12 Phase 5 generators in order:

**Preprocessing (steps 1–9):**
1. Abort if `world.endedBy` is set
2. Compute absolute year from `timeline.startOfTime + years.length`
3. Sum `worldPopulation` from usable cities (before growth)
4. Grow each usable city's population using `REGION_BIOME_GROWTH` biome multipliers
5. Kill/retire illustrates (natural death when `birthYear + yearsActive <= currentYear`; 15% war-related death chance if origin city's country is at war)
6. Propagate religions (single-religion cities: adherence drifts +0.05 toward 0.9; multi-religion cities: random religion gains +0.05 if total < 0.9; recompute member counts)
7. End expired wars (`started + lasts < currentYear`), clear `atWar` flags
8. Reassert `atWar` flags for active wars
9. Recompute `region.hasResources` via `updateHasResources()`

**Event generation (Phase 5, in order):**
Foundation → Contact → Country → Illustrate → Religion → Trade → Wonder → Cataclysm → War → Tech → Conquer → Empire

Each generator produces 0 or 1 event per year and may mutate world state (e.g., founding a city, starting a war, destroying a wonder).

The timeline classes (`Timeline`, `Year`) and all Phase 5 entity instances live **only inside the worker** alongside the physical model — they are not serialized across the `postMessage` boundary. The generator singletons follow the same pattern as the physical model generators.

### Phase 6: HistoryGenerator (Orchestration)

`HistoryGenerator` in `history/HistoryGenerator.ts` is the top-level entry point for the history pipeline. It:

1. Calls `buildPhysicalWorld` to create the `World` with continents, regions, resources, and cities
2. Calls `TimelineGenerator.generate()` to run 5000 years of simulation
3. Builds a `CountryIndexMap` mapping internal string IDs → numeric indices for ownership arrays; generates unique country display names via `generateCountryName()` (50 roots × 24 nation-style suffixes) and empire display names via `generateEmpireName()` (54 title templates, cached per empire)
4. Serializes timeline years into `HistoryYear[]` with `HistoryEvent` objects for each of the 12 Phase 5 event types
5. Computes cell-level ownership snapshots from the region-based country model (countries own regions; conquests transfer region ownership)
6. Converts founded `CityEntity` objects into render-type `City[]`
7. Generates roads via A* between founded cities
8. Produces `HistoryStats` for optional introspection (peak population, entity counts, etc.)

The worker delegates to `historyGenerator.generate()` when `generateHistory` is true. The old `generateHistory()` function in `history.ts` is preserved for backward compatibility but the worker now uses the Phase 6 orchestrator.

### Randomness

All randomness goes through the seeded `mulberry32` PRNG in `terrain/noise.ts`. Never use `Math.random()` directly — pass the seeded RNG to any function that needs randomness to ensure reproducibility.

### Generation Parameters

`GenerateRequest` (in `types.ts`) carries all user-controlled inputs to the worker:

| Parameter | Type | Description |
|-----------|------|-------------|
| `seed` | `string` | Deterministic seed string |
| `numCells` | `number` | Voronoi cell count (500–100,000) |
| `waterRatio` | `number` | Fraction of cells that are water (0–1, default 0.4) |
| `width` / `height` | `number` | Canvas dimensions |
| `generateHistory` | `boolean` | Whether to run the history simulation (default false) |
| `numSimYears` | `number` | Years to simulate (50–5000, default 5000); only used when `generateHistory` is true |
| `profileName` | `string?` | Named **biome** profile preset (e.g. `'desert'`, `'ice'`, `'default'`); defaults to `'default'` which preserves current behavior |
| `shapeName` | `string?` | Named **landmass shape** preset (`'default'`, `'pangaea'`, `'continents'`, `'islands'`, `'archipelago'`); applied as a `Partial<TerrainProfile>` overlay on top of the biome profile; defaults to `'default'` (empty overlay, no-op — the biome profile's baked-in shape hints shine through) |
| `profileOverrides` | `Partial<TerrainProfile>?` | Fine-tuning overrides on top of the named preset; merged via spread (wins over both biome and shape) |

`waterRatio` is implemented by ranking all cells by elevation and marking the lowest `waterRatio * N` as water. This guarantees the exact ratio regardless of the terrain shape, unlike a fixed elevation threshold.

### Rendering

`renderer/renderer.ts` draws everything onto a single `<canvas>` element. Layer visibility is controlled by the `LayerVisibility` type. When modifying rendering:
- Biome colors are defined in `terrain/biomes.ts` (`BIOME_INFO`)
- Coastlines use noisy edges from `renderer/noisyEdges.ts` for an organic look
- City icons are drawn as simple SVG-path-like canvas commands
- `drawBiomeFill` renders land cells first, water cells second — this ensures water always wins at shared polygon edges (Voronoi cell indices have no spatial order, so rendering in index order causes land to bleed over water). Land cells use per-cell vegetation density modulation (`getVegetationDensity` + `modulateBiomeColor`) to vary fill color by ±12% based on moisture position within the Whittaker band, with spatial-hash dither to prevent banding. Tree icons also vary in density (10–30%), size (0.85–1.15×), and color (lighter green at dry edges, darker at wet edges) based on the same factor
- `drawHillshading` computes per-cell elevation gradients from Voronoi neighbors and applies directional illumination (NW light, 315° azimuth, 45° altitude) as an rgba overlay — white for lit slopes, black for shaded slopes. Placed between biome fill and water depth in the render order. Controlled by `layers.hillshading` (defaults to enabled). Uses elevation scale factor of 8.0 to exaggerate relief
- `drawPermafrost` renders a blue-gray overlay (`rgba(180, 200, 220, alpha)`) on sub-polar land cells (temperature 0.10–0.30). Placed between hillshading and water depth. Alpha scales with depth into the cold band and varies by season. Controlled by `layers.seasonalIce`
- `drawBiomeFill` accepts a `season` parameter; when season is non-zero (and `layers.seasonalIce` is enabled), it calls `getSeasonalBiome()` to determine the effective biome color for each cell, shifting polar biome boundaries per season
- `drawWaterDepth` renders ocean depth overlay with ocean current tinting: compares `cell.temperature` to latitude baseline to derive SST anomaly, then shifts rgb channels (warm currents → less blue/more green, cold currents → deeper blue); the effect is subtle and purely visual
- The biome legend is a React overlay component (`Legend.tsx`), not drawn on the canvas; it is controlled by `layers.legend` (part of `LayerVisibility`) and rendered in `App.tsx`
- When `historyData` is present, kingdom borders/fills use `getOwnerAtYear(history, selectedYear, cellIndex)` instead of `cell.kingdom`; city/road/border layers are hidden entirely when no history data exists
- `getOwnerAtYear` finds the nearest decade snapshot ≤ target year, then replays `ownershipDeltas` forward to the exact year
- **Political view** applies a parchment overlay (`rgba(245, 233, 200, 0.55)`) on land cells and uses patterned country/empire fills via `drawPatternedBorders()` with `PatternCache` (diagonal stripes for countries, plaid for empires; see `renderer/patterns.ts`). Terrain view retains the original subtle solid fills (0.12 alpha via `KINGDOM_COLORS_TERRAIN`)

### City Character Generation

`src/lib/citychars.ts` is a UI-only lazy roller that turns the simulation's race / deity / alignment metadata into a deterministic D&D 3.5e PC roster the moment a city is opened in the Details tab. The fantasy character generator (`src/lib/fantasy/`) is the underlying engine — `generatePcCharBiased(level, parentAlignment, rng, { raceWeights?, deityWeights? })` is the biased entry point added in this layer (the original `generatePcChar` keeps its frozen 3-arg signature). Roster size scales with city tier: `small=3, medium=6, large=12, metropolis=24, megalopolis=48`; "dominant bias" (race + deity locked to the country/dominant religion) scales the opposite way: `1.0 → 0.7 → 0.5 → 0.3 → 0.15`. The roster is rolled on demand inside `DetailsTab.tsx::CityDetails` via a `useMemo` keyed on `(charactersOpen, city, country, cityReligionDetails, history.worldSeed)`; ruined cities and worlds with no `worldSeed` short-circuit to an empty array. The tooltip on each row carries HP, abilities, age, height/weight, and starting wealth so the table itself stays narrow inside the 280 px overlay.

The simulation-side metadata feeds three new fields:

- **`Country.raceBias: { primary: RaceType; secondary?: RaceType }`** — picked at country founding in `CountryGenerator.generate` from a PRNG sub-stream `seededPRNG(`${world.seed}_racebias_${country.id}`)`, weighted by the founding region's biome via `BIOME_RACE_WEIGHTS` (forests/temperate → human + elf + halfling, mountains/tundra → dwarf + gnome, deserts → orc + half-orc). 60% chance of a meaningful secondary race; otherwise mono-cultural. The base race-prob snapshot lives in `Country.ts::BASE_RACE_PROB` to avoid a circular import from `fantasy/RaceType.ts`; keep it in sync with `RACE_SPECS[r].prob` whenever weights change there.
- **`Religion.deity: Deity` + `Religion.alignment: AlignmentType`** — bound at religion founding (Path 1) in `ReligionGenerator.generate` from a PRNG sub-stream `seededPRNG(`${world.seed}_deity_${religion.id}`)`, weighted toward deities of the founder country's `raceBias.primary` race (×3) with universal deities at ×1.5 and race-disallowed deities at 0. The display name stays illustrate-derived ("Faith of Aldric Stormvale") and `deity` / `alignment` are separate fields shown alongside.
- **`World.seed: string`** — copied from the worker's seed string by `WorldGenerator.generate(rng, seed)` so simulation generators can derive their isolated sub-streams. Threaded through `buildPhysicalWorld(cells, width, rng, rarityWeights, seed)` and `historyGenerator.generate(cells, width, rng, numSimYears, rarityWeights, seed)`. Defaults to `''` for the sweep harness and any standalone test path; sub-stream draws are still deterministic, just keyed off the empty-string root.

Serialization (`HistoryGenerator.ts`) copies `country.raceBias` straight into `HistoryData.countries[i].raceBias`, builds `religionDetails: ReligionDetail[]` (mirrors `wonderDetails`), `cityReligions: Record<number, string[]>` (final-year cellIndex → religion ids sorted by adherence descending — first id is the dominant religion), and stores `worldSeed: string` so `lib/citychars.ts` can derive its `${seed}_chars_<cellIndex>` PRNG client-side.

**Invariants**:
- **Sweep stability is preserved by design**: race bias and deity binding draws go to ISOLATED PRNG sub-streams (`_racebias_<countryId>`, `_deity_<religionId>`), they do not perturb the main timeline RNG, and neither field enters `HistoryStats`. `npm run sweep` must remain byte-identical against `scripts/results/baseline-a.json` after any change to `Country.ts`'s `pickRaceBias`, `Religion.ts`'s `pickDeity`, or the `BIOME_RACE_WEIGHTS` table. A non-zero diff means a sub-stream draw accidentally leaked into the main `rng` parameter — fix the leak rather than rebaseline.
- **No simulation feedback in v1**: `Country.raceBias` and `Religion.deity / alignment` are purely decorative. They are NOT consumed by `WarGenerator` / `ConquerGenerator` / `TradeGenerator` / `ReligionGenerator.Path 2` / etc. Future work could (e.g. wars more likely between alignment-incompatible deities, trade easier between same-race countries) but doing so would require a sweep rebase.
- **Do NOT import `lib/citychars.ts` from the worker** (`src/workers/mapgen.worker.ts`) or from `src/lib/history/`. The roller is render-side only — it depends on render-layer types (`City`, `Country`, `ReligionDetail`) and rolls fresh PRNGs on demand. Importing from the worker would defeat the lazy-on-open design and balloon the postMessage payload.
- **Do NOT import `fantasy/techNames.ts`-style flavor data into mutation sites**: the same naming-vs-mutation discipline applies — `lib/fantasy/Deity.ts` `DEITY_SPECS` is referenced by `Religion.ts` (sim, picks the deity), `HistoryGenerator.ts` (serialize, reads display name), and `lib/citychars.ts` (UI, picks character deities). No other mutation sites should touch it.
- **`generatePcCharBiased` does not mutate `RACE_SPECS` / `DEITY_SPECS`**: it constructs local biased copies inside the function and discards them at return. Calling it with `{}` (or omitting `opts`) is byte-identical to `generatePcChar`. If you add a new bias dimension (e.g. class), follow the same local-copy pattern — never mutate the shared spec records, or any future call site that runs in parallel will silently observe a stale weight table.
- **`citychars.ts` uses `seededPRNG(`${worldSeed}_chars_${cellIndex}`)`** — independent of every other PRNG sub-stream in the codebase. Same `(worldSeed, cellIndex)` pair always returns the same roster; this is the contract Details-tab code relies on for stable rosters across re-mounts.

## Common Pitfalls

- **Worker communication**: `mapgen.worker.ts` uses `postMessage` with typed `WorkerMessage` objects. Keep the message schema in sync with `App.tsx`'s `onmessage` handler.
- **High-DPI canvas**: `MapCanvas.tsx` scales the canvas by `devicePixelRatio`. Don't set canvas width/height via CSS — use the component's resize logic.
- **Cell count performance**: Generation above ~10,000 cells is slow. Default is 5,000. Test UI changes at low cell counts.
- **Base path**: Local `npm run dev` serves from `/`, but production uses `/procgen_map_fe/`. Avoid hardcoded absolute paths in source.
- **Elevation normalization**: After computing FBM + island-falloff elevations, `terrain/elevation.ts` divides all values by the observed maximum so the highest cell always reaches 1.0. Without this, FBM noise in practice tops out around 0.8, and the island mask compresses it further — leaving the Whittaker mountain band (elevation > 0.75) unreachable. Do not remove this normalization step.
- **Mountain biome bands**: The Whittaker table uses 5 elevation bands: lowland (<0.3), midland (0.3–0.6), highland (0.6–0.65), alpine (0.65–0.75, yields `ALPINE_MEADOW` as a transitional biome), and mountain (0.75+, yields SCORCHED/BARE/TUNDRA/SNOW). Mountain icons are rendered on all land cells with elevation ≥ 0.75 regardless of biome `iconType`, at ~40% density with elevation-scaled sizing.
- **History is the cities/kingdoms source**: Do not call `placeCities` or `drawKingdomBorders` from the worker when `generateHistory` is true — `HistoryGenerator` owns that responsibility. Calling both would double-place cities and corrupt kingdom state.
- **HistoryGenerator is the new orchestrator**: When history is enabled, the worker calls `historyGenerator.generate()` (Phase 6), not the old `generateHistory()` from `history.ts`. The old function is preserved but not used by the worker.
- **Ownership reconstruction**: `getOwnerAtYear` must apply deltas in strict year order. Out-of-order application produces incorrect borders. The snapshots are keyed by decade (0, 10, 20…); always start from `Math.floor(year / 10) * 10`.
- **`cell.kingdom` vs history**: `cell.kingdom` is written once by `history/history.ts` as the year-0 state. The renderer overwrites the visual ownership at the selected year but must never mutate `cell.kingdom` — it is the baseline and is needed to reconstruct history from scratch.
- **`buildPhysicalWorld` always runs**: Unlike the history simulation, `buildPhysicalWorld` is called for every generation (terrain-only or history). `MapData.regions` and `MapData.continents` are always populated. Do not gate region/resource rendering on `mapData.history`.
- **Hydraulic erosion refreshes temperature/biomes**: After `hydraulicErosion` modifies elevations, the worker re-runs `assignTemperature` and `assignBiomes` because they depend on elevation (lapse rate, elevation bands). Moisture and ocean currents are NOT re-run — they depend on coarse geography. Rivers are also re-traced after erosion so they follow the carved terrain.
- **`accumulateFlow` topological order must follow `drainageElev`, not raw elevation**: when `fillDepressions` has run, drainage points *uphill in raw elevation* through every filled basin (pit bottoms drain to their pour point, which has higher `cell.elevation`). If `accumulateFlow` sorts by raw `cell.elevation` descending, the pour point is processed *before* the pit cells and propagates its partial flow downstream; the pit's contribution then gets stranded at the pour point when the pit cells are processed later. Visible symptom: downstream rivers thin or disappear just past a filtered (too-small-to-be-a-lake) depression, even though `drainageElev` routes correctly. The fix is to sort `order` by `drainageElev[idx]` descending when provided — PF+ε makes that surface strictly monotonic along every drainage edge, so a cell is always processed before its drainage target. The bug was latent until `lakeMinSize` started leaving small basins as land; before then, every depression became a lake, lake cells were skipped by both `buildDrainageMap` and `accumulateFlow`, and no uphill-drainage edges existed in the processing graph.
- **Rivers must go through `fillDepressions` or they'll dead-end mid-land**: before any `generateRivers` call the worker MUST run `fillDepressions(cells, profile)` and thread the returned `drainageElevation` into `generateRivers(cells, profile, drainageElev)`. Without the PF+ε pass, closed basins (cells lower than all their neighbours) have `drainage === null`, flow accumulated at those cells never propagates downstream, and the resulting river chains visibly terminate in the middle of land — exactly the bug this pass was introduced to fix. The worker calls `fillDepressions` twice on purpose: once after `assignBiomes` to guide the initial river pass (which in turn drives erosion), and a second time after `hydraulicErosion` because erosion's deposition step (`cells[downstream].elevation += deposit`) can create new sinks that the first pass never saw. Do NOT pass a cached `drainageElev` into `hydraulicErosion` — its per-iteration `buildDrainageMap` must keep using raw `cell.elevation` because the deposition breaks monotonicity and a stale drainage surface would point erosion at the wrong cells. The `isLake` short-circuit in `assignBiomes` (which protects LAKE cells from the post-erosion biome refresh) MUST stay *after* the polar ICE block — moving it earlier would prevent cold inland lakes from freezing (think Lake Baikal in winter). Lakeshore land cells still get `isCoast = true` from `hydraulicErosion`'s coast re-mark pass, so `scoreCellForCity` in `history.ts` treats lakeshores as ocean coast for city placement (+3 score) — this is intentional for v1 (lakeshore cities are realistic and visually rich); if that scoring turns out to bias cities too heavily toward lakes, add an `isLake`-aware exclusion in the coast re-mark loop rather than touching `fillDepressions`.
- **Don't postMessage class instances**: `World`, `Region`, `Continent` use `Map` and `Set` which are not structured-clone safe. Only the plain `RegionData[]`/`ContinentData[]` arrays cross the worker boundary. Keep the class instances inside the worker.
- **TRADE_TECHS duplication**: the trade-tech field list is hardcoded in two places — `TRADE_TECH_FIELDS` in `physical/CityEntity.ts` (consumed by `effectiveTradeCap()`) and `TRADE_TECHS` in `timeline/Tech.ts`. The duplication exists to avoid a `physical → timeline` circular import. If you change one, change the other. A dev-only monotonicity assertion in `mapgen.worker.ts` guards against regressions in the multiplier formula.
- **Phase 1 tech effects use the scope helpers**: `science`, `military`, `energy`, `biology`, `art`, `government`, and `industry` all read tech levels via `getCityTechLevel` / `getCountryTechLevel` (exported from `timeline/Tech.ts`), which mirror the empire-founder → country → city scope ladder used by `_createTech`. Do NOT read `country.knownTechs.get(...)` directly in new code — empire-member countries will silently miss the founder fallback. **Phase 2 throughput note**: `techGenerator.generateForYear` is called **exactly once per year** by `YearGenerator` step 10. It returns up to `N = clamp(0..5, floor(log10(worldPop / 10_000)))` techs by rolling each country independently with chance `min(1, illustrateCount / 5)`, and may append one stateless legacy pick if slots remain. Phase 1 coefficients (e.g. the science weight cap of 3×) were originally tuned for the old `rndSize(5, 1)` loop; under Phase 2 they apply only to the stateless tail.
- **HistoryStats is forwarded, not recomputed**: `historyGenerator.generate()` returns `stats: HistoryStats` and the worker forwards it as `MapData.historyStats`. New code that wants tech aggregates (`totalTechs`, `peakTechLevelByField`, Phase 4's `totalTrades` / `totalConquests` / `totalCataclysmDeaths` / `techEventsPerCenturyByField` / `peakCountryTechLevelByField` / `medianCountryTechLevelByField`, spec stretch §1's `totalTechLosses` / `totalTechLossesAbsorbed`, or spec stretch §2's `totalTechDiffusions`) or peak population should read from `mapData.historyStats` instead of re-walking `historyData.years` on the main thread.
- **Tech-loss writes go through `getCountryEffectiveTechs`**: the spec stretch §1 cataclysm tech-loss path in `Cataclysm.ts` mutates the same `Map<TechField, Tech>` that all reads use, so empire-member countries decrement the founder's shared knownTechs map (correct: an empire shares its knowledge, so a fire in any member region damages the federation's collective record). Do NOT bypass the helper or the founder will silently drift from members. Same rule as the Phase 1 read-side pitfall — both directions go through the scope ladder. **Spec stretch §2** trade-driven tech-diffusion writes follow the same rule via `recordDiffusedTech` in `Tech.ts`: it writes through `getCountryEffectiveTechs(world, receiver)` so an empire-member receiver mutates the founder's shared map. The eligibility check in `Trade.ts` also reads donor/receiver levels via `getCountryEffectiveTechs` for the same reason.
- **Named techs are serialization-only**: spec stretch §3 names live in `src/lib/history/timeline/techNames.ts` and are resolved **exclusively** inside `HistoryGenerator.ts` — the TECH event push, the CONQUEST `acquiredTechs` mapping, and the `buildTechDescription` helper. Never import `techNames.ts` from `Tech.ts`, `Cataclysm.ts`, `Trade.ts`, or any other mutation site. The table is flavor metadata, not simulation state; routing it through the mutation layer would couple balance changes to text edits. The naming change has zero balance impact and must not perturb any sweep metric — a non-zero sweep diff after touching `techNames.ts` means a mutation site was accidentally modified.
- **Religion drift is per-(city × religion), not per-city**: spec stretch §4 splits the step 6 drift computation so each religion in a city can pick up its own `government` bonus from its own `originCountry`. The `art` bonus is still city-scoped (+0.02 when the hosting city has `art` at country scope); the `government` bonus is religion-scoped (+0.01 per level, cap +0.03 at level 3+). `originCountry` is *snapshotted at founding* — a religion founded in country A that later conquers the whole continent still reads A's government level, even if A has been subsumed into an empire (the scope ladder then resolves the empire founder's level transparently via `getCountryTechLevel`). Null `originCountry` (pre-country founding) means both §4 effects silently no-op. Do NOT resolve `originCountry` dynamically from the founding city's *current* `regionId → countryId` — that would steal the bonus on conquest and diverge from the spec's "whose city hosts the religion's *founding* event" language. The Path 2 outward-expansion weighting in `Religion.ts` uses the same snapshotted `originCountry`, so the donor country's institutional strength — not the source city's current owner — drives the bias. At `government = 0` a fast path preserves uniform selection and identical RNG usage to pre-§4 code.
- **Phase 4 tuning constants — re-run the sweep before landing**: when adjusting `growth` (YearGenerator.ts), `exploration` BFS depth (Contact.ts), or the `Tech._throughputCap` formula, run `npm run sweep -- --label <experiment>` and diff against `scripts/results/baseline-a.json`. The Phase 4 quality gates (in `specs/28_Tech.md`) are ±25% on peak/median per-field tech levels, ±30% on totalWars/totalTrades/totalConquests, and ±30% on peakPopulation. The sweep is byte-deterministic, so any non-zero diff is a real behavior change. Current Phase 4 landing point: `growth = 0.12`, `exploration depth = 1 + ceil(level/2)`, throughput unchanged.
- **Sweep harness lives in `scripts/sweep-history.ts`**: it's a single-file Node CLI run via `tsx` (dev dep). It imports the same pure functions the worker does and replays the full pipeline per seed. Do NOT import anything from `src/components/`, `src/workers/`, or any DOM-dependent module from here — the harness must remain browser-free so it can run in CI or in a bare Node shell. If the terrain or history pipeline grows a new step, mirror it in both `mapgen.worker.ts` and `sweep-history.ts` (same order, same arguments) or future sweeps will silently drift from the in-browser behavior.
- **TechTimeline is monotonic and computed inline with the tech stats walk** (spec stretch §5): `HistoryGenerator.ts` populates `techTimeline.byField[f][yi]` in the **same** `timeline.years` loop that updates `peakTechLevelByField` / `techEventsPerCenturyByField`. Do NOT add a second walk — the existing loop already visits every tech event. The inside-loop write is guarded with `if (yi < yearsToSerialize)` because the outer loop iterates `timeline.years.length`, which may exceed `yearsToSerialize` on truncated runs (out-of-bounds `Uint8Array` writes are silently dropped, so skipping this guard would lose tail data). After the loop a cheap per-field forward-fill sweep propagates each year's max forward through quiet years. **Do NOT dip the curves on `TECH_LOSS` events** — the chart intentionally shows "highest peak ever attained per field", not "live current level". If a future revision wants a non-monotonic current-level series, add a second `Uint8Array` field to `TechTimeline` rather than mutating the existing one. The chart is read-side only: touching it must not affect the sweep output. `TECH_FIELD_COLORS` lives in `overlay/eventStyles.ts` (consolidated there in Phase 5 of `specs/overlays_tabs.md`; imported by `TechTab.tsx`) and is declared as `Record<TechField, string>` so adding a new tech field to the `TechField` union in `timeline/Tech.ts` becomes a compile-time error at both the color map and the `HistoryGenerator.ts` `TECH_FIELDS` local constant (line 771) — mirror both sites together.
- **Pattern cache is per-render**: `PatternCache` in `renderer/patterns.ts` is created fresh at the start of each `render()` call (before the 3x horizontal offset loop) and discarded afterward. Do NOT cache it across frames — the `CanvasPattern` objects are tied to the rendering context and may become invalid if the canvas is resized or the context changes. The cache's `getStripe` / `getPlaid` methods lazily create pattern tiles on first access and reuse them within the same frame.
- **River rendering is zoom-aware**: `drawRivers()` in `renderer/renderer.ts` divides `ctx.lineWidth` by the `scale` threaded down from `MapCanvas` because the canvas ctx is already transformed by `transform.scale` before `render()` is called — setting `lineWidth = N` without the divide produces `N * scale` screen pixels, which is why rivers used to look visibly thick at high zoom. Culling uses `visibilityCutoff = RIVER_VISIBILITY_BASE / scale²` against the stored `river.maxFlow`, so the generation-time `profile.riverFlowThreshold` remains the absolute floor (no amount of zooming can reveal rivers that never passed threshold during generation). If you want more/fewer rivers at a given zoom, tune `RIVER_VISIBILITY_BASE`; if rivers look too thin/thick overall, tune `RIVER_MIN_SCREEN_PX` / `RIVER_MAX_SCREEN_PX` / `RIVER_WIDTH_COEFF`. The `Minimap` calls `render()` without a `scale` argument, falling back to the default `scale = 1` — this intentionally leaves the minimap with only trunk rivers visible, which is a cleaner overview than the pre-change behavior of stroking every tributary as sub-pixel lines.
- **SHAPE_PROFILES entries must be `Partial<TerrainProfile>`, never a full profile**: shapes stack as a middle layer in the worker merge (`biomeProfile → shapeOverlay → user profileOverrides`), so every field a shape doesn't explicitly set falls through to the biome layer — that's the entire point of orthogonal biome × shape dimensions. Making a shape a full `TerrainProfile` would silently override every biome knob and collapse the two dimensions back into one. When tuning a shape, only touch the tectonic/shape subset (`numContinentalMin/Max`, `numOceanicMin/Max`, `continentalGrowthMin/Max`, `seamBoostMin/Max`, `seamSpreadRings`, `elevationPower`); never add moisture/temperature/biome-threshold fields to a shape. The `default` shape is deliberately `{}` so biome profiles with baked-in shape hints (e.g. `ocean`'s 0 continental plates) keep producing byte-identical output when the user hasn't picked a shape. The sweep harness (`scripts/sweep-history.ts`) currently uses `DEFAULT_PROFILE` directly with no biome/shape overlay — this is intentional, so sweep baselines stay byte-identical. If the harness ever gains `--profile`/`--shape` CLI flags, mirror the three-way merge from `mapgen.worker.ts` exactly.
- **TerrainProfile threads through every terrain function**: all 6 terrain functions (`assignElevation`, `computeOceanCurrents`, `assignMoisture`, `assignTemperature`, `assignBiomes`, `hydraulicErosion`) accept a `profile: TerrainProfile` as their last parameter. The worker resolves the profile from `GenerateRequest.profileName` + `GenerateRequest.shapeName` + `profileOverrides` and passes it to each call. The sweep harness uses `DEFAULT_PROFILE` directly. When adding a new tunable terrain constant, add it to the `TerrainProfile` interface in `types.ts`, set its default in `DEFAULT_PROFILE` in `profiles.ts`, and read it from the profile parameter — do NOT add a new file-local const. The biome threshold consts (`ICE_TEMP_THRESHOLD`, `SNOW_TEMP_THRESHOLD`, `TUNDRA_TEMP_THRESHOLD`) are kept as file-local consts in `biomes.ts` because `getSeasonalBiome()` (a render-time function) also uses them and the profile only exists in the worker; `assignBiomes` reads from the profile instead. Seven profiles are now defined in `PROFILES`: `default` (Earth-like), `desert` (arid, ~70% desert), `ice` (snowball, ~80% frozen), `forest` (greenhouse, ~75% forest), `swamp` (flat/wet), `mountains` (tectonic, ~40% highland), and `ocean` (archipelago, ~85% water). Each non-default profile spreads `DEFAULT_PROFILE` and overrides only theme-relevant fields. `PROFILE_WATER_RATIOS` provides the recommended `waterRatio` per profile for the UI. `globalMoistureOffset` and `globalTempOffset` are additive shifts applied as the final step of `assignMoisture` / `assignTemperature` respectively — with default value 0.0 they are no-ops, preserving byte-identical output for the `default` profile.
- **Empire snapshots are a replay, not a live read** (overlay Phase 4): `HistoryGenerator.ts` reconstructs empire membership chronologically inside the main year loop by mirroring `Conquer._handleEmpireEffects` — walk `yearObj.conquers` to remove the conquered from its current empire (dissolve at ≤1), then add it to the conqueror's empire; apply `yearObj.empires` as fresh entries with `{founder, triggering-conquer.conquered}`; finally consult a pre-indexed `dissolvedByAbsYear: Map<number, Set<string>>` built from each `Empire.destroyedOn` to catch the invisible 15% `government`-tech dissolution path from `ConquerGenerator`. Do NOT read `CountryEvent.memberOf` or `Empire.countries` at serialization time — those fields reflect the **final** simulation state, not year-i state, so a naive snapshot would bleed the last year's membership backward through every earlier frame. The replay reads `yearObj.conquers` / `yearObj.empires` only and uses `world.mapCountries` / `world.mapRegions` strictly read-only to resolve founder display names; it must never mutate any `World` field, because that's what keeps `npm run sweep` (which only hashes `HistoryStats`, not `empireSnapshots`) byte-identical. When adding new member-transition logic to `ConquerGenerator`, mirror it in the replay block or the Hierarchy tab will silently diverge from reality.
- **Conquest government dissolution reads pre-merge tech levels**: `ConquerGenerator` captures both countries' government tech levels **before** the `mergeAllTechs` call and uses those for the 15% dissolution check. After the merge, the conqueror always has `max(conqueror, conquered)` per field, so reading post-merge would make the check dead code. Do NOT move the government-level capture below the merge.
- **Empire wonder stability reduces dissolution chance**: `ConquerGenerator` computes `getEmpireStandingWonderCount` (from `Wonder.ts`) to reduce the 15% government-tech dissolution probability by 2% per standing wonder across all empire member countries (including expansion regions), floored at 0%. At 8+ empire-wide standing wonders, dissolution from this path is impossible. The wonder count is read after `_handleEmpireEffects` (so the just-conquered country's wonders count toward stability) but before region transfers; wonder state is unaffected by either. The `HistoryGenerator.ts` empire snapshot replay is unaffected — it reads `Empire.destroyedOn` directly and does not re-roll the probability.
- **`_handleEmpireEffects` must do full dissolution**: when an empire shrinks to ≤1 member, `_handleEmpireEffects` must clear `memberOf` on all remaining members, empty `empire.members`/`countries`/`reach`, and set `destroyedOn`/`conqueredBy`. Leaving any of these stale creates "zombie empires" where the remaining member has `memberOf` pointing to a destroyed empire, which blocks `EmpireGenerator` from creating new empires, prevents valid wars via the same-empire check in `War.ts`, and corrupts the tech scope ladder in `getCountryEffectiveTechs`. The correct pattern matches `Ruin.ts _dissolveCountry` (lines 217-225) and `_dissolveEmpire` (lines 128-137 in Conquer.ts).
- **`computeOwnership` must include dead countries**: `computeOwnership()` in `HistoryGenerator.ts` seeds ownership and `regionOwner` from both `world.mapCountries` and `world.mapDeadCountries`. Dead countries (dissolved when all cities become ruins via `ruinifyCity`) still have a valid `governingRegion` string, and historical conquests referencing them must transfer their regions. The conquest replay lookup also checks both maps (`mapCountries ?? mapDeadCountries`). `buildCountryIndexMap` already indexes dead countries, so `idToIndex` lookups succeed for both.
- **War defender must not be at war**: `WarGenerator` checks `!c.atWar` for the aggressor (line 56) and also checks `candidate.atWar` for the defender in the selection loop. Both belligerents must be at peace to enter a new war. This prevents simultaneous multi-front wars and the edge cases they create with `atWar` flag management during conquest resolution.
